# Stage 3.5: Feature Decomposition (MANDATORY CHECKPOINT)

After Stage 3, you have enough information to determine whether this feature should be split into multiple documents. This stage always runs, even if the result is "no split needed."

## 3.5.1 Cluster value streams

Group the Stage 2 actors + goals + Stage 3 variables into natural clusters. A cluster is a group of related actor-goals that share a common user outcome and system dependency chain.

Present as a table:

| Cluster | Core user outcome | Actors involved | Key variables (from Stage 3) | Primary system dependency | Regional / audience scope |
|---|---|---|---|---|---|
| A | [What the user gains] | [Which actors] | [Which variables] | [Which API/system] | [Who cares about this] |
| B | [What the user gains] | [Which actors] | [Which variables] | [Which API/system] | [Who cares about this] |

## 3.5.2 Test each cluster for independence

For each cluster, answer three questions:

| Cluster | Can it ship if others are delayed? | Does it have a unique blocking dependency? | Does it deliver a standalone user outcome? |
|---|---|---|---|
| A | [Yes/No — explain] | [Yes — name it / No — shares with X] | [Yes/No] |

A cluster that answers "no" to any question should be merged into the cluster it depends on.

## 3.5.3 Sizing signals

| Cluster | Projected FRs | Projected user flows | Projected scenarios (rough) | Sizing signal |
|---|---|---|---|---|
| A | [count] | [count] | [count] | [Right-sized / Too large / Too small] |

Thresholds that trigger a recommendation (signals, not hard rules):
- **Consider splitting** when a cluster exceeds 7 FRs, 6 user flows, or 40 projected scenarios
- **Consider merging** when a cluster has fewer than 2 FRs and cannot ship independently
- A small cluster (1-2 FRs) is justified if it has a distinct blocking dependency and can ship on its own timeline

## 3.5.4 Build the Shared Registry

The Shared Registry is a binding contract that prevents duplication and conflict across feature documents. It assigns single ownership for every item that spans multiple features.

**A. Shared FRs — ownership and reference rules**

| FR / Behavior | Description | Owned by | Referenced by | Rule |
|---|---|---|---|---|
| [e.g., Notification framework] | [What it covers] | [Feature A] | [Features B, C] | [Feature A owns framework rules. Others add only their trigger-specific rules and reference Feature A for delivery mechanism.] |

**B. Shared Assumptions — single source of truth**

| Assumption | Owned by | Why this feature | Referenced by |
|---|---|---|---|
| [e.g., WFM union status identification] | [Feature A] | [Core request flow depends on it] | [Feature B] |

Rule: each assumption exists in exactly one feature document. Other features reference it.

**C. Shared Dependencies — single owner**

| Dependency | Owned by | Referenced by |
|---|---|---|
| [e.g., WFM submission API] | [Feature A] | [Features B, C note "requires Feature A infrastructure"] |

**D. Shared Open Questions — single owner**

| Open Question | Owned by | Referenced by |
|---|---|---|
| [e.g., Pat MacArthur backup contact] | [Feature A] | [Features B, C] |

**E. Platform Constraints — single home**

Universal constraints (e.g., channel supplementation, Mingle SPOF, accessibility) live in the primary (largest) feature. Other features add a cross-reference: "See [Feature A], Section N."

## 3.5.5 Present recommendation

State the recommendation in plain English:

> "Based on [N] distinct value streams, [N] independent blocking dependencies, and [N] distinct user outcomes, this feature should be split into [N] documents: [names]. Feature [X] is the core and ships first. Features [Y] and [Z] depend on it but can be delayed independently."

Or:

> "This feature has [N] value streams but they share [dependency] and cannot ship independently. Recommend keeping as a single document."

Or:

> "Cluster [X] is too small to justify a standalone document ([N] FRs, [N] scenarios). Recommend merging into [Y]."

## 3.5.6 Regional applicability table

If the split produces features with different regional scope, include a regional applicability table for each feature:

| Region | What applies | What doesn't |
|---|---|---|
| [Region 1] | [FRs / rules that apply] | [FRs / rules that don't] |

This table goes at the top of each feature's requirements document (Stage 7) so regional stakeholders can immediately filter.

# Execution Model: Staggered Parallel (Multiple Features)

When Stage 3.5 produces a split, Stages 4-9b run per feature using a staggered model:

**Phase 1: Primary feature (full checkpoints)**

The largest / most-depended-on feature runs Stages 4 through 9b first with all normal checkpoints (Stage 5 assumption review, Stage 9a/9b audit). This produces:
- A complete, user-validated requirements document
- Confirmed assumptions that resolve questions likely to appear in smaller features
- A proven pattern for the domain that smaller features can follow

**Phase 2: Remaining features (parallel, reduced checkpoints)**

After the primary feature's Stage 5 is confirmed, remaining features launch in parallel as subagents. Each subagent receives:
1. All shared context (Stages 1-3 artifacts, Stage 3.5 Shared Registry)
2. The primary feature's confirmed Stage 5 assumptions (so they don't re-ask resolved questions)
3. An explicit exclusion list derived from the Shared Registry: "Do NOT generate requirements, assumptions, dependencies, or open questions for the following items. Instead, add a cross-reference to [Feature A]."
4. A flag instruction: "If you discover a new item that applies to multiple features and is not in the Shared Registry, tag it as `[SHARED — assign to Feature X]` and do not resolve it yourself."

Smaller features skip the Stage 5 user-confirmation checkpoint (their assumptions are scoped and the major shared ones are already confirmed). They still run Stage 9a/9b for individual document quality.

**Phase 3: Post-merge reconciliation (Stage 9c)**

After all features complete Stages 4-9b, Stage 9c runs. See `stages/09c-reconciliation.md`.

**Single feature (no split):** The execution model does not apply. Stages 4-9b run sequentially as before.
