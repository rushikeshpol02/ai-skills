# Stage 3.5c: Staggered-Parallel Execution Model

> **Conditional load only.** This file is read only when Stage 3.5b confirms a split. For single-feature runs, skip entirely.

---

## Primary Wave

The largest or most-depended-on feature runs Stages 4–9 first with all normal checkpoints. This produces:
- A complete, PM-validated requirements document
- Confirmed assumptions that resolve questions likely to appear in sibling features
- A proven domain pattern sibling features can follow

## Parallel Wave

After the primary feature's **Stage 5 is confirmed**, launch remaining features as parallel subagents. Each subagent receives:

1. All shared context: Stage 1–3 artifacts + Stage 3.5 Shared Registry
2. The primary feature's confirmed Stage 5 assumptions — sibling features do not re-surface resolved questions
3. An explicit exclusion list from the Shared Registry:
   > "Do NOT generate requirements, assumptions, dependencies, or open questions for the following items. Add a cross-reference to [Feature A] instead: [list]"
4. A flag instruction:
   > "If you discover a new item that applies to multiple features and is not in the Shared Registry, tag it `[SHARED — assign to Feature X]`. Do not resolve it yourself."

**Checkpoint rules for parallel features:**
- Stage 5 (assumptions) user-confirmation checkpoint for **shared assumptions** is **skipped** — major shared ones are already confirmed from the primary wave. Feature-unique HIGH assumptions discovered for this feature must still be surfaced to the PM before Stage 7a. Any HIGH assumption not present in the Shared Registry exclusion list requires PM confirmation before proceeding.
- Stage 9 (validation) runs for every feature — no exceptions

## Reconciliation

After all features complete their per-feature stages, **Stage 9c runs once** across all features. See `09c-reconciliation.md`.

## Execution Order in State File

Update `pipeline-state.json`:
- `split_features` → `["Feature A", "Feature B"]` in execution order
- Primary feature is always index 0
