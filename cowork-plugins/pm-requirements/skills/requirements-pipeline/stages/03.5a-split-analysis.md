## Step 1: Cluster Value Streams

Group Stage 2 actors + goals + Stage 3 variables into natural clusters. A cluster = related actor-goals that share a common user outcome and system dependency chain.

| Cluster | Core user outcome | Actors | Key variables | Primary system dependency |
|---|---|---|---|---|
| A | [What the user gains] | [Which actors] | [Stage 3 variables] | [API/system] |

## Step 2: Independence Test

For each cluster:

| Cluster | Ships if others delayed? | Unique blocking dependency? | Standalone user outcome? |
|---|---|---|---|
| A | Yes / No | Yes — [name] / No — shares with X | Yes / No |

A cluster answering "No" to any question merges into the cluster it depends on.

## Step 3: Safe Split Criteria

A split is safe only when **all four hold:**
1. A PM can review each document without reading the sibling first.
2. No FR in Feature A depends on a condition created by an FR in Feature B.
3. The split follows a natural boundary — user does X or Y (parallel paths), not X then immediately Y.
4. Feature A can ship without Feature B and produce a coherent experience.

**Stop and warn before recommending a split when any danger signal applies:**

| Danger signal | Consequence |
|---|---|
| Features share more than 2 steps of a core user flow | Split creates orphaned flow cards |
| One feature lists the other as a hard dependency | Documents must be read together |
| The boundary is invisible to the user | Both documents describe the same session |
| Sub-features are steps in a workflow (A then B, not A or B) | Neither document is complete standalone |

When a danger signal is present, describe it explicitly and ask for PM direction before continuing.

## Step 4: Sizing Signals

| Cluster | Projected FRs | Projected flows | Sizing signal |
|---|---|---|---|
| A | [count] | [count] | Right-sized / Too large / Too small |

Thresholds (signals, not hard rules):
- **Consider splitting:** cluster exceeds 7 FRs or 6 user flows
- **Consider merging:** cluster has fewer than 2 FRs and cannot ship independently

**If "Too large":** Apply Steps 1–3 recursively to that cluster only. Depth limit: 2 levels. A cluster requiring a third-level split signals scope is not well-defined — present to PM and recommend narrowing scope.

## Step 6: Recommendation

State in plain English:
- **Split:** "Based on [N] distinct value streams and [N] independent blocking dependencies, recommend splitting into [N] documents: [names]. [Feature X] ships first."
- **No split:** "Value streams share [dependency] — cannot ship independently. Recommend single document."
- **Merge:** "Cluster [X] has [N] FRs and cannot ship standalone. Recommend merging into [Y]."

PM must confirm: split decision, feature names, execution order.

## Step 7: Rename Run Folder

After PM confirms split decision, rename the run folder to reflect confirmed feature names:
- Single feature: `_runs/[feature-slug]-[YYYYMMDD]/`
- Multiple features: `_runs/[feature-a]+[feature-b]-[YYYYMMDD]/`

Update `pipeline-state.json`: set `run_id` to the new folder name and update all `artifacts.*` paths.
